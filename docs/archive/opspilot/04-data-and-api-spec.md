# Data & API Specification — OpsPilot

What the prototype consumes today (synthetic SQLite), what production would
integrate (OMS/WMS/carrier APIs), and the contracts for the three LLM
workflows.

## 1. Synthetic dataset design

All data is fictional and generated by `src/data_gen.py` (single seed in
`src/config.py`; dates anchor to run time so the demo is always "today").
Design constraints, in order of importance:

1. **Internal consistency.** The generator creates raw *state* only — inventory
   positions, payment flags, carrier dates. The exception engine then detects
   issues from that state. Every exception in the demo is therefore *true*:
   a STOCKOUT's SKU really cannot cover its order line (asserted in
   `tests/test_engines.py`).
2. **Realistic distributions.** Weekday-weighted order flow; Pareto customer
   concentration (tier-weighted propensity — top accounts drive ~5× the order
   volume of median accounts); long-tail SKU velocity; ~85% home-region DC
   assignment.
3. **Analyzable patterns.** Deliberate signal is embedded for the AI digest to
   find: one carrier (TransCon Freight) causes a majority of transit delays;
   SLA-risk orders cluster at one DC; several stockouts have inbound-PO
   recovery paths while others don't.

Volumes: ~520 orders / 30 days · 60 customers · 80 SKUs · 3 DCs · 240
inventory positions · ~460 shipments · ~75 detected exceptions.

## 2. Data dictionary (8 tables)

Schema: `sql/schema.sql`. Analytical queries: `sql/queries.sql` (named blocks
loaded by `src/db.py`).

**customers** — `customer_id` PK · `name` · `tier` (Platinum/Gold/Standard) ·
`region` · `industry`. Tier drives ship SLA (1/2/3 business days) and scoring.

**products** — `sku` PK · `product_name` · `category` (5 industrial
categories) · `unit_price` · `unit_cost`.

**warehouses** — `warehouse_id` PK (WH-ATL, WH-DFW, WH-RNO) · `name` · `region`.

**inventory** — PK (`sku`,`warehouse_id`) · `on_hand` · `reserved` (allocated
to open orders) · `damaged_qty` (unsellable) · `reorder_point` · `incoming_qty`
· `incoming_eta`. **ATP = on_hand − reserved − damaged_qty** — the single
availability truth used everywhere.

**orders** — `order_id` PK · `customer_id` FK · `warehouse_id` FK · `channel`
(web/edi/phone) · `order_date` · `promised_ship_date` (order date + tier SLA in
business days) · `status` (pending→allocated→shipped→delivered; on_hold;
cancelled) · `payment_status` · `address_valid` · `total_value` ·
`priority_score` (written by the engine).

**order_items** — PK (`order_id`,`line_no`) · `sku` FK · `qty` · `unit_price`
(tier-discounted).

**shipments** — `shipment_id` PK · `order_id` FK · `carrier` · `ship_date` ·
`promised_delivery_date` · `actual_delivery_date` (NULL in transit) · `status`.

**exceptions** — `exception_id` PK · `type` (8 values) · `severity` · `status`
· `order_id`/`sku`/`warehouse_id` (nullable by type) · `owner_team` · `title` ·
`description` (carries concrete figures for the AI context) · `created_at` ·
`resolved_at`.

## 3. Production integration requirements (proposed)

The prototype reads batch state; production replaces the generator with:

**Inbound (system → OpsPilot):**

| Source | Interface | Payload → table |
|---|---|---|
| OMS | webhook `order.created`, `order.updated`, `payment.status_changed` | orders, order_items |
| WMS | webhook `inventory.adjusted`, `shipment.created`; hourly snapshot reconcile | inventory, shipments |
| Carrier APIs | polling/webhook tracking scans | shipments (delivery dates, scan staleness) |
| Address validation | synchronous at intake | orders.address_valid |

**Outbound (OpsPilot REST API, read-only in pilot):**

```
GET  /api/v1/queue?warehouse=WH-ATL&limit=50      # scored open orders
GET  /api/v1/orders/{order_id}/score              # component breakdown
GET  /api/v1/exceptions?status=open&severity=critical
GET  /api/v1/exceptions/{id}                      # detail + context
POST /api/v1/exceptions/{id}/recommendations      # run workflow 2
POST /api/v1/copilot/answers                      # {question} → workflow 3
GET  /api/v1/digest?date=today                    # workflow 1 (cached daily)
POST /api/v1/exceptions/{id}/resolve              # {resolution_note} — pilot's only write
```

Auth: service-to-service tokens; per-user attribution deferred to R6
(write-back), which additionally requires an audit log and idempotency keys.
Detection cadence: event-driven per webhook + 15-min sweep reconciliation.

## 4. LLM workflow specifications

Shared contract — all three workflows:

- **Pipeline:** gather context (SQL) → retrieve SOP sections → compose.
- **Grounding guardrails** (system prompt, `src/llm/prompts.py`): use only
  supplied data; never invent IDs/quantities/dollars/policies; cite SOP
  sections; state insufficiency instead of guessing.
- **Modes:** live (Anthropic API, model configurable via `LLM_MODEL`, default
  `claude-sonnet-5`) or offline (deterministic composition over the *same*
  context — identical layout, zero cost). Mode is always visible in the UI.
- **Retrieval:** hand-rolled TF-IDF over ~40 SOP sections chunked by `##`
  heading (`src/retrieval.py`). Deliberate pilot tradeoff: at this corpus
  size, classic retrieval is accurate with zero infra. The `search()`
  interface is the seam for an embedding upgrade (roadmap R4).

**Workflow 1 — Daily digest.** Input: KPI snapshot, exceptions by
type/severity, top-5 by severity×value, carrier delay concentration, SLA-risk
by DC, value at risk. Output contract: headline → top 3 risks (with EXC/ORD
IDs) → pattern watch → 3 per-team focus actions with SOP citations.
Live budget: ~1k tokens out. Cache: once per day is sufficient.

**Workflow 2 — Recommend actions.** Input: exception row + type-specific
context (ATP across all DCs, `qty_required`, inbound POs, shipment scans,
duplicate pair, customer tier/value) + top-k SOP sections for the type.
Output contract: situation (concrete numbers) → 2–4 ordered actions, each
with rationale + SOP citation, *only* options the data allows → escalation
check. Hard requirement carried by both modes: if no transfer cover exists,
transfer must not be proposed.

**Workflow 3 — SOP Q&A.** Input: user question + top-3 retrieved sections.
Output: direct answer → conditions/thresholds → citations; explicit "SOPs
don't cover this, ask <team>" when retrieval confidence is empty. Offline
mode returns governing policy text verbatim with citations (honest, not
imitation-AI).

## 5. Non-functional notes

**Latency:** offline <200ms; live dominated by one LLM call (~2–5s) — accepted
for pilot; digest is cacheable. **Cost:** ≤1 digest + ~30 recommendations +
~20 questions/day ≈ low single-digit USD/day at Sonnet-class pricing; $0
offline. **Privacy:** synthetic data end-to-end; production prompts would carry
order/customer fields — requires DPA review before pilot go-live (tracked in
pilot plan risks). **Failure behavior:** any API error degrades to offline
composition — the product never blocks on the LLM.
